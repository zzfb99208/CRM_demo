# CRM Demo 标准化开发流程

> 每次新增功能时，复制以下内容发给 Claude Code，按步骤执行。

---

## 使用方法

将下面的提示词完整复制给 Claude Code，替换 `[你的功能描述]` 为实际需求：

```
我需要对 CRM Demo 系统新增一个更新：有两个bug需要修改1.现在无法通过审核在审核员端点通过审核会出现Approval failed: null报错分析原因并解决2.现在在pi编辑阶段上下调的按钮被遮挡了显示不全并且没有和数据库的库存信息联动起到限制作用超出库存没有提示（理想情况是可以超出但是保存时候需要有二次确认提示） 

请严格按照以下 4 个阶段顺序执行，每一阶段完成后暂停，等我确认再继续：

---

## 阶段 1：需求分析 & PRD 更新

1. 阅读 `docs/design/CRM_产品需求文档.md` 了解当前功能边界
2. 根据我的功能描述，分析影响范围：
- 涉及哪些角色（业务员/审核者）？
- 涉及哪些页面和操作流程？
- 需要新增 API 还是修改现有 API？
- 是否需要改数据库（新增表/字段）？
3. 更新 `docs/design/CRM_产品需求文档.md`，将新需求写入对应章节
4. 输出的 PRD 变更内容让我确认

---

## 阶段 2：设计文档同步更新

基于阶段 1 确认的 PRD 变更，依次更新：

1. `docs/design/CRM_数据库设计.md` — 新增表/字段的 DDL + LLM 提示词更新
2. `docs/design/CRM_Java后端设计.md` — 新增/修改 API + 伪代码 + LLM 提示词更新
3. `docs/design/CRM_Vue前端设计.md` — 新增/修改页面布局 + 组件 + LLM 提示词更新
4. 输出 3 份设计文档的变更汇总，让我确认

---

## 阶段 3：代码实现

基于阶段 2 确认的设计，按此顺序修改代码：

1. 数据库 — 执行 DDL 变更（建表/加字段/种子数据）
2. 后端 — Entity → Mapper → Service → Controller（先逻辑后接口）
3. 前端 — API 层 → Store → 页面组件（先数据后视图）
4. 每一步完成后编译验证，确保无错误
5. 全部完成后 `git add -A && git commit && git push`

---

## 阶段 4：审核 & 维护

1. 输出本次改动汇总表（改了什么文件、新增了什么功能）
2. 更新 `启动指南.md` 中的操作流程（如果流程有变化）
3. 给出验证步骤让我自测
4. 记录已知问题和后续优化方向
```

## 代码修改顺序（备忘）

每次改代码时按这个优先级：

```
数据库 DDL
    ↓
Entity（实体字段）
    ↓
Mapper（无代码，MyBatis-Plus 自动）
    ↓
Service（业务逻辑）
    ↓
Controller（API 端点）
    ↓
编译验证 → mvn compile
    ↓
前端 API 层（api/*.js）
    ↓
前端页面（views/*.vue）
    ↓
前端路由（router/index.js，如需要）
    ↓
编译验证 → npx vite build
    ↓
Git 提交推送
```

## Git & GitHub 版本管理命令速查

### 日常开发

```bash
# 查看当前状态（改了什么文件）
git status

# 查看具体改了什么内容
git diff

# 暂存所有修改
git add -A

# 提交（-m 后面写修改说明）
git commit -m "描述你的修改"

# 推送到 GitHub
git push

# 一键三连（暂存 + 提交 + 推送）
git add -A && git commit -m "描述" && git push
```

### 分支操作

```bash
# 查看所有分支（本地 + 远程）
git branch -a

# 基于当前代码创建新分支
git checkout -b 分支名

# 推送到 GitHub 并建立追踪
git push -u origin 分支名

# 切换分支
git checkout 分支名

# 切回 master
git checkout master

# 删除本地分支
git branch -d 分支名
```

### 备份与回滚

```bash
# 在大改动前创建稳定备份分支
git checkout -b stable-v2
git push -u origin stable-v2
git checkout master    # 回到 master 继续开发

# 查看提交历史
git log --oneline -10

# 回滚到某个提交（保留修改）
git reset 提交ID

# 回滚到某个提交（丢弃修改，危险操作）
git reset --hard 提交ID

# 从备份分支恢复 master
git checkout master
git reset --hard stable-v1
git push --force-with-lease
```

### 查看远程信息

```bash
# 查看远程仓库地址
git remote -v

# 查看本地和远程的差异
git log master..origin/master --oneline
```

### 典型场景速查

| 场景 | 命令 |
|------|------|
| 新功能开始前备份 | `git checkout -b stable-vN && git push -u origin stable-vN && git checkout master` |
| 每日收工提交 | `git add -A && git commit -m "WIP" && git push` |
| 功能完成提交 | `git add -A && git commit -m "Add xxx feature" && git push` |
| 改坏了想回滚 | `git checkout -- 文件名`（单文件）/ `git reset --hard HEAD`（全部） |
| 推送前看看远程有啥新的 | `git fetch && git log HEAD..origin/master --oneline` |
